## What to read

1. `README.md` (and `START-HERE.md`, `SETUP.md`, `QUICKSTART.md`, `DEVELOPERS.md`, `CONTRIBUTING.md` if present)
2. `package.json` / `pyproject.toml` / `go.mod` / `Cargo.toml` (whatever package manifest applies)
3. The top-level directory tree — understand the layout
4. Recent commit history (`git log --since='90 days ago' --oneline --no-merges`)
5. Any schema files: `prisma/schema.prisma`, `migrations/`, `*.sql`
6. Any API spec: `openapi.yaml`, `swagger.json`, or route handlers
7. CI config: `.github/workflows/`, `vercel.json`, etc.
8. Deploy/infra config: `Dockerfile`, `terraform/`, `pulumi/`, env files

For each file with TODO markers, replace them with real content:

- One-sentence "what this repo is and who uses it"

## What to leave alone

- Generated files (`docs/STACK.md`, `DATA.md`, `API.md`, `MAP.md`, `INTEGRATIONS.md`) — they're regenerated by `scripts/llm-docs/gen-*.sh`. Don't touch them.
- Frontmatter blocks at the top of every file — keep them.
- Navigation headers (`> **Anchor:** [↑ ARCHITECTURE.md]...`) — keep them.
- `research/` and `scripts/llm-docs/` — these are template infrastructure.
- The "See also" footers at the bottom of each file.

## Quality bar

- **Be specific.** Reference real file paths, real function names, real env vars.
- **Be honest.** If something is broken / missing / TODO, say so. The whole point of these docs is to give agents accurate ground truth.
- **Don't speculate.** If you can't determine something from reading the code, leave it as a smaller TODO with a note about what to look for.
- **Cross-reference.** Every file should link to ARCHITECTURE.md as the anchor. The spine should feel like a connected graph, not isolated files.
